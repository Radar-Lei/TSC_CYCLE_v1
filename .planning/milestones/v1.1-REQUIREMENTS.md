# Requirements: TSC-CYCLE v2

**Defined:** 2026-04-02
**Core Value:** 让模型在严格输出格式和绿灯约束下，稳定学会按输入饱和度分配合理的相位绿灯时间

## v1 Requirements

### Reward Format

- [x] **FORM-01**: 开发者可以对简化版 GRPO completion 是否包含 `<end_working_out><SOLUTION>...</SOLUTION>` 结构进行 reward 打分
- [x] **FORM-02**: 开发者可以复用现有 solution JSON 协议，使简化版 GRPO 继续输出按相位顺序排列的 `{"phase_id": int, "final": int}` 列表

### Constraint Guard

- [x] **CONS-01**: 开发者可以对 completion 的 phase 顺序、solution 长度、`final` 整数类型进行约束校验
- [x] **CONS-02**: 开发者可以对每个 phase 的 `final` 执行 `min_green <= final <= max_green` 校验，并将越界 completion 置为低分或零分

### Saturation Reward

- [x] **SATR-01**: 开发者可以从 prompt 中解析 `phase_waits[*].pred_saturation`、`min_green`、`max_green`，为每个 phase 计算饱和度比例目标绿灯时间
- [x] **SATR-02**: 开发者可以在满足最小绿约束的前提下，用 completion 的 `final` 与目标绿灯时间的接近程度计算 reward，而不依赖 SUMO 仿真

### Training Pipeline

- [x] **PIPE-00**: 开发者可以通过 Unsloth Docker 生成和训练简化版 GRPO 数据与模型，而不是依赖宿主机环境
- [x] **PIPE-01**: 开发者可以在独立的新目录中运行简化版 GRPO 训练入口，而不覆盖现有 `src/grpo/` 完整版实现
- [x] **PIPE-02**: 开发者可以将简化版 GRPO 的 checkpoints 和最终模型输出到独立目录，便于与现有 `outputs/grpo/` 区分
- [x] **PIPE-03**: 开发者可以直接从 `outputs/data/train.jsonl` 生成简化版 GRPO prompt 数据，而不依赖 GLM-5 SFT 样本

### Verification

- [x] **VERI-01**: 开发者可以通过基础单元测试验证格式错误、约束越界、比例命中和比例偏离这几类 reward 行为

## v2 Requirements

### Evaluation

- **EVAL-01**: 开发者可以将简化版 GRPO 模型接回 benchmark 流程做离线效果评估
- **EVAL-02**: 开发者可以比较简化版 GRPO 与 SUMO reward GRPO 的训练结果差异

### Advanced Reward

- **ARWD-01**: 开发者可以在饱和度比例 reward 之外，再叠加 queue、delay 或 throughput 等多目标 reward
- **ARWD-02**: 开发者可以探索不同的 target 归一化或全周期预算分配策略

## Out of Scope

| Feature | Reason |
|---------|--------|
| SUMO 仿真 reward | 本里程碑目标就是移除仿真依赖 |
| baseline 预计算 | 简化版 reward 不再依赖 baseline.json |
| 直接改写 `src/grpo/` | 用户要求新建目录与旧版并存 |
| 默认复用 GLM-5 SFT 数据做简化版 GRPO 数据源 | 新 reward 目标与 SFT 数据分布可能不一致 |
| Benchmark 指标结论 | 先把简化版训练闭环做出来，评估后置 |
