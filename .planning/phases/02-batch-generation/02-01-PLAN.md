  <action>
    创建 `src/glm5/validator.py`，实现输出解析和约束校验:

    1. 定义 ParsedOutput dataclass:
       ```python
       @dataclass
       class ParsedOutput:
           success: bool
           think_text: str = ""        # <start_working_out> 和 <end_working_out> 之间的内容
           solution: list = field(default_factory=list)  # JSON 数组
           think_length: int = 0       # think_text 的字符数 (粗略 token 估计)
           raw_content: str = ""       # 原始内容
           error: str = ""             # 失败原因
       ```
       包含 to_dict() 方法。

    2. 实现 parse_glm5_output(content: str) -> ParsedOutput:
       - 用正则提取 think 链: 匹配 `<start_working_out>(.*?)<end_working_out>` (re.DOTALL)
         如果内容没有 <start_working_out> 前缀，则 think_text 为 <end_working_out> 之前的全部文本
       - 用正则提取 solution: 匹配 `<SOLUTION>(.*?)</SOLUTION>` (re.DOTALL)
       - 对 solution 文本做 json.loads()
       - 验证结果是 list
       - 计算 think_length = len(think_text) (字符数，粗略 token 近似)
       - 任一步骤失败返回 ParsedOutput(success=False, error=具体原因)

    3. 实现 validate_constraints(solution: list, phase_waits: list[dict]) -> tuple[bool, str]:
       核心逻辑提取自 src/grpo/rewards.py check_constraints (lines 183-230):
       - 检查 len(solution) == len(phase_waits)，不等则返回 (False, "相位数量不匹配: 期望{N}个, 实际{M}个")
       - 检查相位顺序: solution[i]["phase_id"] == phase_waits[i]["phase_id"]，不一致则返回 (False, "相位顺序不一致: 期望{expected}, 实际{actual}")
       - 检查每个相位: final 必须为 int 类型 (isinstance(final, int))，否则返回 (False, "phase_id={X} final 必须为整数, 实际为{type}")
       - 检查每个相位: phase_waits[i]["min_green"] <= final <= phase_waits[i]["max_green"]，否则返回 (False, "phase_id={X} 绿灯时间{final}越界 [{min}, {max}]")
       - 全部通过返回 (True, "")

    4. 遵循项目惯例: 中文 docstring, Google-style, 双引号, `__all__`
