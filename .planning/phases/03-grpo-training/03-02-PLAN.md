Create src/grpo/rewards.py (~250-300 lines) containing all reward functions for GRPO training.

**Architecture:** Follow the reference script pattern (qwen3_(4b)_grpo.py lines 272-391) -- multiple independent reward functions that are passed to GRPOTrainer.reward_funcs as a list. Each function receives `completions` (and optionally `prompts`, `**kwargs`) and returns a list of scores.

**Tag format:** `<think>...</think><CyclePlan>...</CyclePlan>` (per CONTEXT.md decisions and SFT training consistency). The model is prompted to start with `<think>`, so reward functions should check for `</think>` and `<CyclePlan>...</CyclePlan>`.

**Function signatures must match trl GRPOTrainer interface:**
```python
def reward_func(completions, **kwargs) -> list[float]
# completions: list of list of dicts, e.g. [[{"role": "assistant", "content": "..."}], ...]
# For functions needing prompts: def reward_func(prompts, completions, **kwargs)
```

**1. match_format_exactly(completions, **kwargs) -> list[float]**
- Regex: Match `</think>` followed by `<CyclePlan>` content `</CyclePlan>` with optional trailing whitespace/EOS
- If full match: score = config reward.format_exact_score (default 3.0)
- Else: score = 0

Pattern (compile at module level):
```python
import re
match_format = re.compile(
    r"</think>\s*<CyclePlan>(.+?)</CyclePlan>\s*$",
    flags=re.DOTALL
)
```

**2. match_format_approximately(completions, **kwargs) -> list[float]**
- Count tag occurrences in response:
  - `</think>` appears exactly 1 time: +0.5, else -1.0
  - `<CyclePlan>` appears exactly 1 time: +0.5, else -1.0
  - `</CyclePlan>` appears exactly 1 time: +0.5, else -1.0
- No need to check `<think>` since it's always prepended by add_generation_prompt
- Return sum of individual scores

**3. check_constraints(prompts, completions, **kwargs) -> list[float]**
- Extract CyclePlan JSON content from response using match_format regex
- If cannot extract: return -2.0
- Parse JSON: must be a list of objects with phase_id and final fields
- Check constraints (per user decisions -- gradual scoring, not binary):

  a) Phase order correctness:
     - Extract expected phase_ids from the prompt's phase_waits
     - Compare output phase_ids against expected
     - Score = (number of correct positions / total phases) * constraint_phase_order_weight
     - If lengths don't match: 0 for this component

  b) Green time range:
     - For each phase, check min_green <= final <= max_green
     - Score = (number satisfying / total phases) * constraint_green_range_weight
     - Also check final is integer

  - Total score = phase_order_score + green_range_score
  - If either is 0 (not fully satisfied), set a flag in kwargs or module-level state indicating L2 not fully passed

**4. sumo_simulation_reward(prompts, completions, **kwargs) -> list[float]**
- Only executes SUMO when L1 format matches AND L2 constraints ALL satisfied (100% pass rate)
- Gate check: Re-parse response. If format invalid OR any constraint violated, return 0.0
- Extract phase plan from CyclePlan JSON
- Get state_file and tl_id from kwargs (passed via dataset metadata columns)
- Load baseline from module-level cached baseline.json (loaded once at init)
- For each completion that passes gate:
  - Use multiprocessing to run SUMO simulation:
    1. Start SUMO with correct sumocfg (use get_sumocfg_for_state from baseline.py)
    2. loadState the state file
    3. Execute model's plan: for each phase in order, setPhase + step for `final` seconds
    4. After all phases, collect metrics: passed_vehicles and queue_vehicles
    5. Close SUMO
  - Normalize against baseline:
    - throughput_ratio = model_passed / max(baseline_passed, 1)
    - queue_ratio = baseline_queue / max(model_queue, 1)  (lower queue = higher ratio)
    - combined = sumo_throughput_weight * throughput_ratio + sumo_queue_weight * queue_ratio
    - score = min(combined, 1.0) * sumo_max_score
  - If SUMO crashes or times out: raise error (per user decision: terminate program on SUMO system errors)

**Important implementation detail for SUMO simulation:**
Use a helper function `_run_sumo_evaluation(state_file, sumocfg, tl_id, plan, timeout)` that:
- Starts its own SUMO process (unique port via random)
- Uses traci labeled connection (label=f"reward_{os.getpid()}_{port}")
- Has a timeout of 60 seconds real time
- Returns {passed_vehicles, queue_vehicles}

The reward function itself calls this via multiprocessing Pool or directly in process (per user decision: process pool with num_generations workers, but since reward functions are called per-batch, implement as sequential within the function but each call is one process).

Actually, since GRPOTrainer calls reward functions with the full batch of num_generations completions at once, and we want parallel SUMO: use `concurrent.futures.ProcessPoolExecutor` with `max_workers=num_generations` inside the reward function. Cache the pool at module level.

**5. think_length_reward(completions, **kwargs) -> list[float]**
- Extract think content between `<think>` (added by generation prompt) and `</think>`
- If no `</think>` found: return think_penalty
- Tokenize think content (use a simple word count approximation: len(text) / 2 as rough token count, or better: pass tokenizer reference)
- Actually simpler: use character count / 2 as rough token estimate for Chinese text
- If think_tokens < think_min_tokens: penalty = think_penalty * (1 - think_tokens / think_min_tokens)
- If think_tokens > think_max_tokens: penalty = think_penalty * (think_tokens / think_max_tokens - 1)
- If in range: score = 0.0 (no bonus, no penalty)
- Return penalty scores

Structure:
```bash
#!/usr/bin/env bash
set -euo pipefail

################################################################################
# GRPO 训练脚本 - 通过 Docker 容器执行 GRPO 强化学习训练
#
# 前置条件:
#   - outputs/sft/model/ 存在 (SFT 训练后的模型)
#   - outputs/grpo/grpo_train.jsonl 存在 (GRPO 训练数据)
#   - outputs/grpo/baseline.json 存在 (基准仿真结果)
#
# 输出: outputs/grpo/model/* (GRPO 训练后的完整模型)
#       outputs/grpo/checkpoints/* (训练过程中的检查点)
#
# 用法:
#   ./docker/grpo_train.sh
#   ./docker/grpo_train.sh --config custom.json
################################################################################

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
PROJECT_DIR="$(cd "${SCRIPT_DIR}/.." && pwd)"

IMAGE_NAME="qwen3-tsc-grpo:latest"
CONTAINER_WORKDIR="/home/samuel/SCU_TSC"

echo "=========================================="
echo "GRPO 训练阶段 (Docker)"
echo "=========================================="
echo "[项目目录] ${PROJECT_DIR}"
echo "[Docker 镜像] ${IMAGE_NAME}"
echo ""

# 创建输出目录
mkdir -p "${PROJECT_DIR}/outputs/grpo/model"
mkdir -p "${PROJECT_DIR}/outputs/grpo/checkpoints"

# 验证前置条件
SFT_MODEL="${PROJECT_DIR}/outputs/sft/model"
GRPO_DATA="${PROJECT_DIR}/outputs/grpo/grpo_train.jsonl"
BASELINE="${PROJECT_DIR}/outputs/grpo/baseline.json"

if [ ! -d "${SFT_MODEL}" ] || [ ! -f "${SFT_MODEL}/config.json" ]; then
    echo "[错误] SFT 模型不存在: ${SFT_MODEL}"
    echo "请先运行 ./docker/sft_train.sh"
    exit 1
fi

if [ ! -f "${GRPO_DATA}" ]; then
    echo "[错误] GRPO 训练数据不存在: ${GRPO_DATA}"
    echo "请先运行 GRPO 数据生成脚本"
    exit 1
fi

if [ ! -f "${BASELINE}" ]; then
    echo "[错误] 基准仿真结果不存在: ${BASELINE}"
    echo "请先运行 ./docker/grpo_baseline.sh"
    exit 1
fi

echo "[前置检查] SFT 模型: 存在"
echo "[前置检查] GRPO 数据: $(wc -l < "${GRPO_DATA}") 条记录"
echo "[前置检查] 基准结果: 存在"
echo ""

# 通过 Docker 容器执行 GRPO 训练
echo "[开始] GRPO 训练..."
docker run --rm \
    --gpus all \
    --shm-size=32GB \
    --user "$(id -u):$(id -g)" \
    -v "${PROJECT_DIR}:${CONTAINER_WORKDIR}:rw" \
    -w "${CONTAINER_WORKDIR}" \
    -e SUMO_HOME=/usr/share/sumo \
    -e HF_HOME="${CONTAINER_WORKDIR}/.cache/huggingface" \
    --entrypoint python3 \
    "${IMAGE_NAME}" \
    -m src.grpo.train \
        --config config/config.json \
        "$@"

echo ""
echo "=========================================="
echo "[完成] GRPO 训练完成"
echo "=========================================="
echo "[模型输出] outputs/grpo/model"
echo "[检查点目录] outputs/grpo/checkpoints"
echo ""
```
