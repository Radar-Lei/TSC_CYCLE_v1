  <action>
创建 Python 脚本 src/scripts/generate_sft_data.py，功能如下：

1. 读取 outputs/sft/sampled_100.jsonl 中的每条样本
2. 对每条样本，根据 prediction.phase_waits 中的数据，程序化生成：

   **think 内容（中文，50-200 token）：**
   - 分析每个相位的 pred_saturation 大小关系
   - 简要说明分配思路（如"phase X saturation 最高应多分配绿灯时间"）
   - 语言风格：中英混合，专业术语可用英文（per user decision）
   - 示例格式参考 sample_prompt_result.md 中的 RAW 部分
   - think 结构：逐相位分析 -> 简要总结分配思路
   - 不需要数值计算过程（per user decision），只需定性方向分析

   **final 值生成逻辑：**
   - 基于 pred_saturation 按比例在 [min_green, max_green] 范围内分配
   - 具体策略：saturation 越高的相位分配越接近 max_green，saturation 为 0 的相位分配 min_green
   - 简单线性映射：如果所有相位 saturation 都为 0，都给 min_green
   - 如果只有一个相位有 saturation > 0，该相位给 max_green，其余给 min_green
   - 一般情况：按 saturation 比例在 min-max 范围内线性分配
   - final 必须为整数秒（round 取整）
   - 约束校验：确保 min_green <= final <= max_green（clamp）

   **solution 内容：**
   - JSON 数组格式：[{"phase_id": <int>, "final": <int>}, ...]
   - 数组长度等于 phase_waits 长度
   - phase_id 与 phase_waits 一致，顺序一致

3. 组装为 SFT 训练格式。每条数据结构：
   ```json
   {
     "messages": [
       {"role": "system", "content": "<系统提示>"},
       {"role": "user", "content": "<原始 prompt 中去掉系统角色行的部分>"},
       {"role": "assistant", "content": "<think>...<think><solution>[...]<solution>"}
     ]
   }
   ```

   注意格式细节（per user decision 和 SFT-03）：
   - 使用 `<think>...<think>` 和 `<solution>...<solution>` 标签（不是 `</think>` `</solution>`，是重复开标签作为关闭标签）
   - system content = "你是交通信号配时优化专家。"
   - user content = 原始 prompt 减去第一行系统角色行（即从【cycle_predict_input_json】开始）
   - assistant content = `<think>{think_text}<think><solution>{solution_json}<solution>`

4. 双重校验（per user decision）：
   - 程序生成时确保约束
   - 生成后再次校验每条数据的 final 值满足 min_green <= final <= max_green
   - 校验 phase_id 顺序和数量一致
   - 校验 think 长度在合理范围（非空且不过长）

5. 输出到 outputs/sft/sft_train.jsonl
6. 打印统计：总条数、think 平均长度、约束违反数（应为 0）

命令行接口：
```
python3 -m src.scripts.generate_sft_data --input outputs/sft/sampled_100.jsonl --output outputs/sft/sft_train.jsonl
```

注意：think 内容由此脚本程序化生成（模拟 AI 手写效果），不需要调用外部 LLM API。生成的文本应该自然、多样化（可使用模板 + 随机化描述用词），但风格一致。
