**1. 更新 config/config.json:**

在 `training.grpo.reward` 中新增压缩函数配置字段：
```json
"sumo_compression_function": "log1p",
"sumo_compression_steepness": 1.0
```
- `sumo_compression_function`：压缩函数类型，当前唯一支持值为 `"log1p"`（即 `log(1+x)`）。预留字段以便未来扩展其他函数（如 `"sqrt"`, `"tanh"` 等）。
- `sumo_compression_steepness`：压缩陡度参数，用于 `log(1 + steepness * x)` / `log(1 + steepness)`。默认 1.0 等同于当前行为 `log(1+x)/log(2)`。

在 `training.grpo.data_filter` 中新增字段：
```json
"min_samples_threshold": 1000
```
这个阈值用于训练前检查：过滤后数据集必须至少有 1000 条样本才允许训练。

**2. 更新 src/grpo/rewards.py — 压缩函数可配置化:**

在 `sumo_reward` 函数中，将硬编码的压缩逻辑替换为从 config 读取：

当前硬编码（约第 553-555 行）：
```python
compressed = math.log(1 + raw_score)
score = compressed / math.log(2) * sumo_max_score
```

替换为：
```python
compression_fn = _config.get("sumo_compression_function", "log1p")
steepness = _config.get("sumo_compression_steepness", 1.0)

if compression_fn == "log1p":
    compressed = math.log(1 + steepness * raw_score)
    normalizer = math.log(1 + steepness)
    score = compressed / normalizer * sumo_max_score
else:
    raise ValueError(f"Unknown compression function: {compression_fn}")
```

这样做的好处：
- steepness=1.0 时行为与现在完全一致（`log(1+x)/log(2)`）
- 未来可以通过添加 elif 分支支持新的压缩函数，无需改训练流程
- config 中不存在这些字段时有合理默认值，向后兼容

同时更新 `sumo_reward` 函数的 docstring（约第 386-388 行的伪代码部分），使其反映可配置的压缩函数。

**3. 创建 src/scripts/analyze_grpo_training.py:**

训练分析脚本，从训练日志中提取关键指标。脚本接收训练日志文件路径作为参数，输出分析报告。

功能要求：
- **解析训练日志**：GRPOTrainer 每步输出 JSON 格式的训练指标到 stdout（由 tee 捕获到日志文件）。需要从日志中提取每步的 reward 相关字段。
  - GRPOTrainer 日志格式示例：`{'loss': 0.5, 'reward': 1.2, 'reward_std': 0.3, ...}`
  - 关键字段：`reward`（均值）、`reward_std`（标准差）、还有各 reward 函数的独立分数

- **Zero-std 统计**：统计 `reward_std == 0`（或极小，< 0.01）的步数占总步数的比例。这是核心改进指标，v1.0 约 20%，期望 v1.1 显著降低。

- **Reward 分布统计**：计算所有步的 reward 均值、标准差、分位数（10/25/50/75/90）、最小最大值。

- **Reward 趋势**：将训练步按 10% 分段（如前 10%、10-20%...），计算每段的 reward 均值，展示是否有稳步提升。

- **输出格式**：
  - 终端打印中文报告
  - 同时保存到文本文件（参数指定或默认 `outputs/grpo/grpo_analysis.txt`）
  - 报告格式：
    ```
    ==========================================
    GRPO 训练分析报告
    ==========================================

    [训练概况]
    总训练步数: N

    [Zero-std 统计]
    Zero-std 步数: X / N (Y%)

    [Reward 分布]
    均值: X.XX
    标准差: X.XX
    分位数: 10%=X.XX, 25%=X.XX, 50%=X.XX, 75%=X.XX, 90%=X.XX
    最小值: X.XX, 最大值: X.XX

    [Reward 趋势]
    步骤 0-10%: 均值 X.XX
    步骤 10-20%: 均值 X.XX
    ...
    步骤 90-100%: 均值 X.XX
    ==========================================
    ```

- **CLI 接口**：
  ```
  python -m src.scripts.analyze_grpo_training --log outputs/grpo/grpo_train.log --output outputs/grpo/grpo_analysis.txt
  ```

- **日志解析策略**：训练日志是混合内容（有普通打印文本和训练指标行）。使用正则匹配包含 `'reward'` 和 `'reward_std'` 的行，尝试解析为 dict（可能需要处理 Python dict 格式，如单引号）。对无法解析的行静默跳过。

- **鲁棒性**：如果日志为空或无法解析任何训练步数据，输出警告信息而非崩溃。

  <name>Task 2: 创建 grpo_pipeline.sh 端到端流水线脚本</name>
  <files>docker/grpo_pipeline.sh</files>
  <action>
创建 `docker/grpo_pipeline.sh`，这是整个 GRPO 流程的单一入口脚本。

**脚本结构（5 步流水线）：**

```
步骤 1/5: 数据生成（--skip-data 跳过）
步骤 2/5: 数据过滤（--skip-filter 跳过）
步骤 3/5: Baseline 计算（--skip-baseline 跳过）
步骤 4/5: GRPO 训练（--skip-train 跳过）
步骤 5/5: 结果分析（--skip-analysis 跳过）
```

**参数解析：**
- `--skip-data`：跳过 GRPO 数据生成
- `--skip-filter`：跳过数据过滤
- `--skip-baseline`：跳过 baseline 计算
- `--skip-train`：跳过训练
- `--skip-analysis`：跳过结果分析
- 不接受其他自定义参数（纯配置驱动，所有训练参数从 config.json 读取）

**日志管理（每步独立日志文件，覆盖模式）：**
- `outputs/grpo/grpo_generate.log`
- `outputs/grpo/grpo_filter.log`
- `outputs/grpo/grpo_baseline.log`
- `outputs/grpo/grpo_train.log`
- `outputs/grpo/grpo_analysis.log`
- 每步输出使用 `2>&1 | tee "$LOG_FILE"` 同时显示终端和写入日志

**步骤 1: 数据生成**
- 终端打印：`[1/5] 生成 GRPO 数据...`
- 直接调用 `./docker/grpo_data.sh`（不是在 pipeline 内部起 Docker 容器，而是调用现有脚本）
- **注意**：grpo_data.sh 已经处理 Docker 容器逻辑，pipeline 只需调用它并捕获输出到日志
- 但因为 grpo_data.sh 自身也会打印到终端，所以用 `./docker/grpo_data.sh 2>&1 | tee "$LOG_FILE"` 方式

**步骤 2: 数据过滤**
- 终端打印：`[2/5] 过滤数据...`
- 2a. 调用 `./docker/filter_data.sh --skip-baseline`（只做过滤，产生 `outputs/grpo/grpo_train_filtered.jsonl`）
- 2b. filter_data.sh 返回后，pipeline 脚本执行覆盖逻辑（用过滤后数据替换原始文件）：
  - 先备份原始文件：`mv outputs/grpo/grpo_train.jsonl outputs/grpo/grpo_train_original.jsonl`（仅在首次时，如果 _original 不存在）
  - 再覆盖：`cp outputs/grpo/grpo_train_filtered.jsonl outputs/grpo/grpo_train.jsonl`
  - 打印覆盖信息：过滤后样本数、原始样本数

**步骤 3: Baseline 计算**
- 终端打印：`[3/5] 计算 Baseline...`
- 调用 `./docker/grpo_baseline.sh`（直接基于被覆盖后的 grpo_train.jsonl 计算 baseline）
- baseline.json 直接覆盖（baseline 脚本本身就是覆盖写入）

**训练前检查（步骤 3 和 4 之间）：**
- 终端打印：`[检查] 训练前置条件验证...`
- 在 shell 层实现以下检查（汇总所有检查结果后再决定是否停止）：
  1. **文件存在性**：grpo_train.jsonl 和 baseline.json 必须存在
  2. **最少样本数**：`wc -l < grpo_train.jsonl` >= config 中的 min_samples_threshold（用 python 一行命令从 config.json 提取阈值）
  3. **Baseline 完整性**：baseline.json 中的场景数 >= grpo_train.jsonl 中的唯一场景数（用 python 一行命令检查）
  4. **Reward 配置合法性**：sumo_throughput_weight + sumo_queue_weight + sumo_delay_weight 总和为 1.0（用 python 一行命令检查）
- 检查失败行为：汇总所有检查结果，全部打印后再 exit 1（这样用户能看到所有问题而非只看到第一个）

**步骤 4: GRPO 训练**
- 终端打印：`[4/5] GRPO 训练...`
- 调用 `./docker/grpo_train.sh --skip-validate`（跳过训练脚本内置的 reward 验证，因为 pipeline 已经做了更全面的检查）
- 日志文件：`outputs/grpo/grpo_train.log`

**步骤 5: 结果分析**
- 终端打印：`[5/5] 分析训练结果...`
- 通过 Docker 容器调用 `python3 -m src.scripts.analyze_grpo_training --log outputs/grpo/grpo_train.log --output outputs/grpo/grpo_analysis.txt`
- Docker 容器配置参照 grpo_data.sh 的模式（IMAGE_NAME, CONTAINER_WORKDIR, 挂载等）

**脚本头部和尾部：**
- 头部打印：流水线名称、时间戳、各步骤跳过状态
- 尾部打印：完成信息、总耗时、输出文件列表
- `set -euo pipefail` 保证 fail-fast
- 记录开始时间 `START_TIME=$(date +%s)`，结束时计算总耗时

**关键实现注意事项：**
- pipeline 脚本不自己起 Docker 容器运行各步骤。它调用现有的 docker/*.sh 脚本，只有分析步骤（因为没有现成脚本）需要自己起容器
- 由于 `set -e`，各 docker/*.sh 如果内部 exit 非 0，pipeline 会自动 fail-fast
- 脚本需要 `chmod +x`
